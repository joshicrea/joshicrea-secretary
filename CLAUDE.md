# AI秘書 プラグイン

## 開発者向けメモ

このリポジトリは `joshicrea-secretary` Claude Code プラグインです。

## AI秘書のロジック

AI秘書のメインロジックは `.claude/rules/秘書.md` に定義されています。
install.ps1 がこのファイルをユーザーの `~/.claude/rules/秘書.md` にコピー（パス置換済み）します。

## 開発時の注意

- パス参照は必ず `{{SECRETARY_BASE_DIR}}/xxx` の形式で記述する（相対パス禁止）
- コミット前に `python3 ツール/パス検証.py` を実行してパス漏れがないか確認する
- install.ps1 を変更したら `powershell -ExecutionPolicy Bypass -File test_install.ps1` でテストする

## ユーザーシグナル → ナレッジ・仕組み変換（自動・例外なし・2026-05-17確定）

**ユーザーから「疑問・確認・違和感・否定・修正指示・エラー報告」のいずれかを受けたら、それは私の判断や成果物が意図と食い違ったシグナル。即座に深掘り分析と仕組み実装を自動実行する。**

### トリガー（これらが来たら必ず作動）

| パターン | 例 |
|---|---|
| 確認・疑問 | 「本当に大丈夫？」「これでいいの？」「正しい？」 |
| 違和感の提示 | 「なんで〇〇なの？」「〇〇じゃない？」 |
| 代替案の示唆 | 「〇〇のほうがいいのでは？」 |
| 否定・修正 | 「違う」「ダメ」「〇〇に直して」 |
| エラー・品質不合格 | メール下書きの内容ズレ・タスク記録漏れ・Gmail/Calendar連携エラー |

### 強制プロトコル

1. **原因の所在を判定**: 直前のスキル出力・私の判断のどちらが原因か
2. **最低10段階・最大100段階の「なぜ」分析**: 3段階で止めない。仕組みで直せる根本原因まで掘る
3. **根本原因を3点以内に特定**: 「気をつける」で済むものは根本原因ではない
4. **仕組みの実装**（同レスポンス内で完了）:
   - ナレッジ追記: `{{SECRETARY_BASE_DIR}}/memory/学習ログ/YYYY-MM-DD.md`
   - 動作ルール追記: `{{SECRETARY_BASE_DIR}}/動作ルール.md`（ユーザー側で書き換え可能・セッション開始時に自動Read）
   - 必要ならプラグイン本体への変更提案を出力: 「次回プラグイン更新時に `.claude/rules/秘書.md` 該当箇所に反映する」と明記
5. **1〜3行で報告**: 何を仕組み化したかをファイルパスごと示す

### ナレッジ追記フォーマット

```markdown
## YYYY-MM-DD: {問題の要約}
- 事実: {何が起きたか}
- 10段階分析: {最深層の根本原因まで・要約3〜5行}
- 根本原因: {3点以内・仕組みで直せるレベル}
- 仕組み実装: {どのファイルに何を追記したか}
```

これはユーザーの指示がなくても毎回自動で行う。1滴もこぼさない。

### 動作ルール.md（ユーザー書き換え可能領域）

`{{SECRETARY_BASE_DIR}}/動作ルール.md` はユーザーが直接編集できるカスタムルール置き場。プロトコル発動時に自動追記される。セッション開始時の必須チェックで自動Read。

### 共通プロトコル参照

`{{SECRETARY_BASE_DIR}}/ナレッジ/_共通プロトコル.md` に教科書例と動作ルール.mdの初期テンプレ案を持つ。初回購入者の「こういう仕組みが回っている」理解の素地。
